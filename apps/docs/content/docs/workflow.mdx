## Agent Loop

```text
START
  -> Spec Agent
  -> Human spec checkpoint
  -> ODIN
  -> Front Agent + QA Agent
  -> Curator
  -> retry, human checkpoint, next story, or completion
```

The graph is intentionally cyclic. It allows Horus to recover from incomplete frontend work, weak QA evidence, or Curator rejection without pretending the first LLM response is production-ready.

## Specialist Roles

| Agent | Responsibility | Key Output |
| --- | --- | --- |
| Spec Agent | Convert user story into technical specification | `Spec` |
| ODIN | Route work and retries | Routing decision |
| Front Agent | Propose frontend/source changes | `CodeChangeSet` |
| QA Agent | Produce validation expectations and evidence | QA result |
| Curator | Validate implementation against spec and evidence | Pass/fail report |

## State Carried Through the Graph

The workflow state should stay explicit and serializable. Important state categories include:

| State Category | Examples | Why It Matters |
| --- | --- | --- |
| User input | User story, selected workspace, selected project | Keeps the run traceable to the original request |
| Spec output | Requirements, components, states, rules, test expectations | Defines what Front, QA, and Curator must satisfy |
| Agent results | Front code, QA output, inspected files, notes | Preserves each specialist contribution |
| Validation state | Curator verdict, score, missing items, fix target | Drives retry routing |
| Runtime evidence | Preview smoke result, command evidence, event timeline | Prevents purely textual approval |
| Control state | Iteration count, checkpoint status, routing decision | Prevents infinite loops and supports resume |

<Warning>
Any state added to the graph must be safe to persist. Do not add raw provider keys, local absolute paths, terminal-private data, or non-portable process handles.
</Warning>

## Workflow Modes

| Mode | Starts At | Purpose |
| --- | --- | --- |
| `spec_generation` | Spec Agent | Generate or update specs from user stories |
| `chat_code_change` | ODIN | Apply chat-driven project changes |
| `project_construction` | ODIN | Build generated project workspaces from selected specs |

## Routing Semantics

ODIN does not simply call every agent on every loop. It decides which specialist should act based on current state and Curator feedback.

| Situation | Expected Route |
| --- | --- |
| No approved spec exists | Spec generation path |
| Spec approved, no implementation exists | Front and QA |
| Curator says frontend is incomplete | Front, optionally QA if test expectations also changed |
| Curator says QA is weak | QA, optionally Curator after evidence refresh |
| Both code and evidence are weak | Front and QA |
| Retry budget is exhausted | Human checkpoint instead of another automatic loop |

## Validation Loop

<Steps>
  <Step title="ODIN routes work">
    ODIN decides whether Front, QA, or both should run.
  </Step>
  <Step title="Specialists produce evidence">
    Front proposes code changes. QA produces test expectations and runtime evidence where available.
  </Step>
  <Step title="Curator validates">
    Curator compares the original spec, proposed changes, QA output, runtime evidence, and quality gates.
  </Step>
  <Step title="Retry or complete">
    Failed work routes back to ODIN while retries remain. Exhausted retries escalate to human approval.
  </Step>
</Steps>

## Curator Validation Inputs

The Curator should evaluate more than generated code text:

| Input | Validation Question |
| --- | --- |
| Original spec | Did the implementation satisfy the accepted requirements? |
| Front output | Are file operations coherent, scoped, and applicable? |
| QA output | Are tests/checks aligned with requirements and realistic to run? |
| Code change set | Were changes applied under the allowed project root? |
| Preview evidence | Did the generated UI start and respond at a basic level? |
| Workflow events | Does the run history support the reported status? |
| Runtime LLM settings | Was the correct provider/model profile used for the run? |

The pass/fail verdict must produce a concrete fix target: `front`, `qa`, or `both`. Ambiguous feedback makes retry routing unreliable.

## Human Checkpoints

Horus uses checkpoints when:

- A generated spec needs explicit approval or rejection.
- Retry limits are reached and the system needs a human decision.
- A resumable workflow must wait for user input.

File mode persists checkpoints under `HORUS_DATA_DIR`. Postgres mode stores checkpoints in the configured database.

## Loop Safety

Every cyclic workflow needs hard stops:

- Track iteration count in workflow state.
- Record every Curator rejection and routing decision as an event.
- Stop automatic retries when the configured limit is reached.
- Preserve enough state for a human to resume intentionally.
- Prefer explicit "blocked" or "needs human decision" status over silent loops.

This is not only a cost control concern. It protects local previews, generated workspaces, and event logs from unbounded writes.

## Evidence Surfaces

| Evidence | Source |
| --- | --- |
| Workflow events | Persistent workflow event stream |
| Runtime validation | QA preview smoke validation and quality gates |
| Code changes | `CodeChangeSet` records |
| Project commands | Safe CLI runner evidence |
| Preview state | Preview session timeline |
| Agent run map | `/api/agent-runs` snapshots |

<Check>
The workflow is designed to make success auditable. A persuasive agent response is not enough; Horus needs state, evidence, and validation gates.
</Check>

## Failure Handling

| Failure | Expected Behavior |
| --- | --- |
| Provider request fails | Persist failure event and surface actionable provider/settings message |
| Skill file cannot load | Fail the agent step with enough context to repair local skill packaging |
| Preview cannot start | Preserve preview timeline evidence and allow retry after configuration fix |
| Code operation violates write policy | Reject the operation before modifying files |
| Checkpoint cannot persist | Fail loudly; resumability is a workflow guarantee |
| Curator score is below threshold | Route to the required specialist while retry budget remains |
