## Scope

The server exposes an Express API. This page documents the route groups and operational contract at a level useful for frontend engineers, integration work, and debugging.

<Info>
Exact request and response schemas should be read from `packages/shared` and the route implementation when making code changes. This page explains route ownership and runtime behavior.
</Info>

## Route Groups

| Route Group | Purpose | Persistence / Runtime Impact |
| --- | --- | --- |
| `/health` | Non-secret liveness check | No durable write |
| `/api/llm` | Provider profile and credential APIs | Credential store and non-secret LLM profile metadata |
| `/api/workflow` | Start, resume, status, and retry decisions | Workflow state, LangGraph checkpoints, event log, code change sets |
| `/api/workspace` | Workspace folders, user stories, specs, and revisions | Workspace repository |
| `/api/chat` | Generic chat sessions and messages | Chat memory repository |
| `/api/horus` | Horus chat turns and intent routing | Chat memory, preview state, workflow state when chat triggers work |
| `/api/preview` | Preview projects, sessions, devices, timeline, drafts | Preview session repository and runtime preview process state |
| `/api/agent-runs` | Run-flow snapshots and event stream replay | Workflow event log and in-memory SSE stream |
| `/api/project-construction` | Generated project construction runs | Project construction repository and generated workspace metadata |
| `/api/project-files` | Safe project tree/read/write APIs | Project construction repository and bounded project filesystem access |
| `/api/events` | Workflow event stream | Persistent event log plus SSE delivery |

## Streaming Routes

| Stream | Route Family | Use |
| --- | --- | --- |
| Workflow events | `/api/events` | Subscribe to workflow progress and persisted event replay |
| Agent run events | `/api/agent-runs` | Drive the visual run-flow map and replay from a sequence cursor |
| Preview events | `/api/preview` | Surface preview lifecycle and timeline changes |

SSE consumers should track the latest sequence they processed and reconnect with the correct cursor when the route supports it.

## Request Rules

- Validate request bodies against shared schemas or route-local validation before mutation.
- Do not accept file paths that escape the allowed project root.
- Do not send provider secrets to frontend clients.
- Do not mark workflow success from HTTP status alone; use persisted workflow state and Curator evidence.
- Prefer explicit error responses over ambiguous empty objects.

## Error Expectations

| Failure | Expected API Behavior |
| --- | --- |
| Invalid request body | Return a client error with actionable validation details |
| Missing workflow/project/session | Return a not-found response instead of an empty successful object |
| Provider configuration missing | Return a settings/credential error without exposing secret values |
| Preview command blocked | Return a policy/runtime error and preserve preview timeline evidence |
| Persistence failure | Fail loudly; do not pretend a workflow or checkpoint persisted |

## Change Checklist

When adding or changing a route:

- Update the owning route file under `apps/server/src/infrastructure/http/routes`.
- Update shared schemas in `packages/shared` when the route crosses package boundaries.
- Update the frontend API client when the web app consumes the route.
- Add tests for valid input, invalid input, and the relevant persistence behavior.
- Update this page if the route changes public behavior.
