## Security Model

Horus needs provider credentials to run LLM-backed workflows, but those credentials must remain outside public state and generated artifacts.

| Boundary | Rule |
| --- | --- |
| Frontend | Never receives raw provider API keys |
| Server | Resolves credentials from environment or credential store |
| Workflow state | Stores provider references and non-secret metadata only |
| Event logs | Must not contain raw credentials, database URLs with passwords, or local secret material |
| Generated workspaces | Must not include runtime credentials unless explicitly authored by the user as project code |

## Credential Sources

| Source | Use |
| --- | --- |
| Environment variables | Primary source for local development, CI, containers, and deployments |
| Backend credential store | Stores provider profile material for the running backend |
| Runtime LLM settings | Selects provider/model/profile for a workflow thread |

`HORUS_SECRET_KEY` should be configured when relying on persistent local credential storage. A stable key avoids losing the ability to read previously stored credential material across restarts or environments.

## Redaction Rules

Secrets must not be written to:

- Workflow state.
- Workflow events.
- Preview timeline entries.
- Chat messages generated by the backend.
- Curator reports.
- QA evidence.
- Generated project manifests.
- Public docs, examples, screenshots, or issue templates.

When logging configuration, prefer provider names, model names, profile IDs, and whether a credential exists. Do not log credential values.

## Local Development

Recommended local flow:

1. Copy `.env.example` to `.env`.
2. Add provider credentials to `.env`.
3. Keep `.env` untracked.
4. Use the LLM settings UI only for profiles that are safe for the current machine.
5. Reset local credential/state directories only after stopping the server.

## Public Repository Hygiene

Public repository content should contain source, safe defaults, documentation, examples, and tests. It should not include:

- Provider API keys.
- Database URLs with passwords.
- Local runtime state.
- Generated build output.
- Private operator notes.
- Machine-specific absolute paths.

## Review Checklist

Before publishing or merging changes:

- Search for provider key names and obvious secret values.
- Confirm `.env.example` contains placeholders only.
- Confirm generated docs and README do not expose local-only operator filenames.
- Confirm workflow/event changes do not serialize raw credentials.
- Confirm deployment configuration injects secrets at runtime rather than baking them into artifacts.
