## Modes

| Mode | Enable With | Storage |
| --- | --- | --- |
| File | `PERSISTENCE_DRIVER=file` | JSON files under `HORUS_DATA_DIR` |
| Postgres | `PERSISTENCE_DRIVER=postgres` | Database tables and Postgres checkpoints |

Persistence is selected at server startup by `createRepositories()`. The selected repository set is then injected into HTTP routes, use cases, the workflow orchestrator, preview services, event streams, and LangGraph checkpointer creation.

<Info>
The goal is not "local-only" persistence. The goal is portable persistence: file mode for simple local development, Postgres mode for durable shared environments, and no hidden dependency on one machine.
</Info>

## File Mode

Default:

```bash
PERSISTENCE_DRIVER=file
HORUS_DATA_DIR=.horus/data
```

File mode stores:

- Workflow state
- Workspace folders, user stories, specs, and revisions
- Chat sessions and messages
- Frontend project registry data
- Preview sessions, timelines, and drafts
- Code change sets
- Workflow events
- Project construction metadata
- Generated project workspaces
- LangGraph checkpoints
- Redacted LLM profile metadata and encrypted secret storage

<Warning>
File-mode state is local runtime data. It must not be committed.
</Warning>

## File-Mode Directory Ownership

With the default `HORUS_DATA_DIR=.horus/data`, runtime data is organized under the configured data root. Exact subdirectories may evolve, but ownership should remain clear:

| Directory Type | Owner | Typical Contents |
| --- | --- | --- |
| Workflows | Workflow state repository | Thread state, status, agent results |
| Workspace | Workspace store | Folders, stories, specs, revisions |
| Chat memory | Chat memory store | Chat sessions and messages |
| Frontend projects | Frontend project registry | Registered preview project metadata |
| Preview sessions | Preview session store | Session state, device, process timeline |
| Code change sets | Code change repository | Proposed/applied operations |
| Workflow events | Event log repository | Agent events, status transitions, validation evidence |
| Project construction | Project construction repository | Generated project runs and workspace metadata |
| LangGraph checkpoints | Checkpointer | Resumable graph checkpoints |
| Credentials | Credential store | Local provider profile material |

Any new file-mode repository should receive a directory from runtime config instead of computing paths independently.

## Postgres Mode

```bash
PERSISTENCE_DRIVER=postgres
DATABASE_URL=postgresql://user:password@host:5432/horus
DATABASE_SSL=false
```

Postgres mode:

- Creates a database pool.
- Runs migrations on repository creation.
- Uses Postgres-backed repository classes.
- Uses the LangGraph Postgres checkpointer.

## Postgres Runtime Ownership

Postgres mode is intended for environments where state must outlive one local checkout or be shared across deployments. It should own:

| State | Required Behavior |
| --- | --- |
| Workflow state | Persist status, agent outputs, retries, and resumable run data |
| Workspace artifacts | Persist stories/specs and preserve revision history |
| Chat memory | Persist conversations independently from frontend sessions |
| Preview session metadata | Persist logical session state; process handles remain runtime-local |
| Code change sets | Persist auditable file-operation proposals/results |
| Workflow events | Preserve chronological event stream |
| Project construction | Persist generated project metadata and status |
| LangGraph checkpoints | Use database-backed checkpointer |

Process-local resources, such as spawned preview process handles, should not be serialized as durable database data. Persist identifiers, statuses, URLs, ports, and timeline evidence instead.

## Repository Contract Rules

All persistence implementations should follow the same behavioral contract:

- Reads return structured entities, not raw filesystem or database records.
- Writes are atomic enough to avoid corrupting JSON/state files on interrupted runs.
- Missing records produce explicit not-found behavior instead of ambiguous empty objects.
- Secrets are redacted from externally visible entities.
- Repository methods should not depend on the current working directory unless runtime config already resolved it.
- Tests should cover file-mode behavior and any Postgres behavior that differs semantically.

## Local State Reset

Stop the server first, then remove the configured file-mode runtime directory. The commands below use the default local path; replace it if your `.env` sets a different `HORUS_DATA_DIR`.

macOS/Linux:

```bash
rm -rf .horus/data
```

Windows PowerShell:

```powershell
Remove-Item -Recurse -Force .horus/data -ErrorAction SilentlyContinue
```

<Warning>
Resetting local state removes workflows, chats, previews, generated workspaces, and local provider profiles.
</Warning>

## Backup and Migration Notes

File mode is easy to inspect but not a substitute for managed backup. For local development, backing up `HORUS_DATA_DIR` is enough to preserve the local runtime state. For team or deployment environments, prefer Postgres with normal database backup policies.

When moving state between machines:

| Source | Destination | Recommended Path |
| --- | --- | --- |
| File mode | File mode | Copy the configured `HORUS_DATA_DIR` only after stopping the server |
| File mode | Postgres | Use an explicit migration script; do not point Postgres repositories at JSON files |
| Postgres | Postgres | Use database-native dump/restore or managed backups |
| Postgres | File mode | Treat as unsupported unless a dedicated exporter exists |

## Cross-Platform Constraints

Persistence code must be valid on macOS, Windows, and Linux:

- Use Node path utilities instead of slash-concatenated paths.
- Avoid assumptions about drive letters, home directories, Desktop paths, or `/tmp`.
- Keep reset commands documented separately for POSIX shells and PowerShell.
- Avoid committing generated state snapshots as fixtures unless they are minimal, sanitized, and intentionally stable.
- Do not rely on file permission behavior that only exists on one OS.

## Publication Rules

Public repositories should not include local runtime state, secrets, generated build output, dependency directories, or private operator notes. Keep persistence examples focused on configurable runtime roots and documented reset procedures, not on private local filenames.
