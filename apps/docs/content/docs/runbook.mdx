## Command Matrix

| Task | Command |
| --- | --- |
| Install | `pnpm install` |
| Develop all apps | `pnpm dev` |
| Build all apps | `pnpm build` |
| Test main suite | `pnpm test` |
| Type-check | `pnpm type-check` |
| Server only | `pnpm --filter @u-build/server dev` |
| Web only | `pnpm --filter @u-build/web dev` |
| Docs only | `pnpm --filter @u-build/docs dev` |

## First-Triage Sequence

When the app does not run, triage in this order:

<Steps>
  <Step title="Confirm package installation">
    ```bash
    pnpm install
    ```
  </Step>
  <Step title="Check repository-wide build health">
    ```bash
    pnpm build
    ```
  </Step>
  <Step title="Run tests">
    ```bash
    pnpm test
    ```
  </Step>
  <Step title="Start only the failing boundary">
    ```bash
    pnpm --filter @u-build/server dev
    pnpm --filter @u-build/web dev
    ```
  </Step>
</Steps>

Avoid debugging the full stack first when a package-specific command can isolate the failing boundary.

## Health Check

```bash
curl http://localhost:3000/health
```

Expected response:

```json
{ "status": "ok" }
```

The health endpoint must not expose secrets, provider settings, filesystem paths, database URLs, or feature flags that would reveal deployment internals.

## Runtime Verification

For a local smoke test after startup:

| Check | Command | Expected |
| --- | --- | --- |
| API health | `curl http://localhost:3000/health` | JSON status `ok` |
| Docs home | `curl http://localhost:3002/docs` | HTML response |
| Docs LLM index | `curl http://localhost:3002/llms.txt` | List of documentation pages |
| Web frontend | Open Vite URL printed by `pnpm --filter @u-build/web dev` | Product shell renders |

If a command prints a different port, trust the command output instead of assuming a fixed default.

## Common Failures

<AccordionGroup>
  <Accordion title="Provider key is missing">
    Configure `.env` or save a provider profile in the UI. Build can pass without provider keys, but real LLM calls need credentials. Check the selected provider, global model, per-agent overrides, and whether a profile reference is being resolved for the workflow thread.
  </Accordion>
  <Accordion title="Port is already in use">
    Set `PORT` for the API server or let Vite choose another frontend port. Keep docs on `3002` unless you intentionally change the docs command.
  </Accordion>
  <Accordion title="CORS blocks the frontend">
    Set `CORS_ORIGIN` when frontend and backend run on different origins. Use a comma-separated list for multiple local frontend ports.
  </Accordion>
  <Accordion title="Workflow cannot resume">
    Check file checkpoints under `HORUS_DATA_DIR/langgraph-checkpoints` or Postgres checkpoint rows. Also inspect workflow events for the last persisted routing decision.
  </Accordion>
  <Accordion title="Preview cannot start">
    Check command catalog, cwd policy, preview port, package manager, and preview timeline evidence. A preview process is runtime-local even when preview metadata is durable.
  </Accordion>
  <Accordion title="File-mode state appears missing">
    Confirm `HORUS_DATA_DIR` is the same value used in the previous run. Relative paths are resolved from the current process context, so container and host runs should set the path intentionally.
  </Accordion>
  <Accordion title="Postgres starts but data is empty">
    Confirm `PERSISTENCE_DRIVER=postgres`, the correct `DATABASE_URL`, and whether migrations ran against the expected database.
  </Accordion>
  <Accordion title="Docs build fails">
    Run `pnpm --filter @u-build/docs build` directly. Most docs failures are invalid MDX, missing imports, or broken generated source from Fumadocs.
  </Accordion>
</AccordionGroup>

## Evidence to Capture in Issues

When reporting a failure, include:

- Operating system and shell.
- Node and pnpm versions.
- Command that failed.
- Full error message.
- Whether the run used file mode or Postgres mode.
- Relevant non-secret environment values such as `PERSISTENCE_DRIVER`, `HORUS_DATA_DIR`, `PORT`, and `CORS_ORIGIN`.
- Last visible workflow event or preview timeline item.

## Reset Local Runtime State

Stop the server first, then remove the configured file-mode runtime directory. The commands below use the default local path; replace it if your `.env` sets a different `HORUS_DATA_DIR`.

macOS/Linux:

```bash
rm -rf .horus/data
```

Windows PowerShell:

```powershell
Remove-Item -Recurse -Force .horus/data -ErrorAction SilentlyContinue
```

Do not reset local runtime state while the server is running. Stop the server first so file writes and checkpoint writes cannot interleave with deletion.
